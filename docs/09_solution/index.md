Архитектурное решение для платформы унифицированных коммуникаций (Unified Communications, UC) описывает целевую микросервисную архитектуру, предназначенную для корпоративного использования, с учётом требований по безопасности, масштабируемости, наблюдаемости и интеграции с внешними системами.

### Архитектурные допущения и границы ответственности

В рамках данной архитектурной работы принимаются следующие допущения (assumptions) и устанавливаются границы ответственности (boundaries):

- **Организационный контур**
  - Платформа UC предназначена для использования средними и крупными корпоративными заказчиками (1–50 тыс. активных пользователей).
  - Эксплуатация платформы может осуществляться как в облачной среде (private/public cloud), так и в дата-центре заказчика; базовый вариант архитектуры ориентирован на развёртывание в Kubernetes-кластере.

- **Функциональный охват платформы**
  - В архитектурное решение входят следующие домены и микросервисы: `Identity/IAM`, `API Gateway`, `SIP Gateway/SBC`, `WebRTC Gateway`, `Call Control`, `Presence`, `Messaging/Chat`, `Media/Recording`, `Provisioning`, `Billing/CDR`, `Notification`, `Observability`.
  - Интеграция с корпоративными системами (AD/LDAP, HR, CRM, ERP, Service Desk) рассматривается на уровне высокоуровневых интерфейсов и протоколов, без детальной проработки схем данных конкретных вендоров.

- **Нефункциональные критерии**
  - Целевой SLA по доступности пользовательских коммуникационных сервисов: не ниже 99.9 % при отказе одного узла в пределах кластера.
  - Платформа должна масштабироваться горизонтально за счёт увеличения количества реплик микросервисов и узлов кластера.
  - В качестве базового уровня безопасности используются: OAuth2.1/OIDC для пользовательской и сервисной аутентификации, mTLS между внутренними сервисами, шифрование трафика (TLS) на внешних границах.

- **Технические ограничения и стандарты**
  - Для сигнализации голосовых вызовов используется протокол SIP, для браузерных клиентов — WebRTC (SRTP/DTLS).
  - Для обмена событиями и асинхронных интеграций используется брокер сообщений (например, Kafka или RabbitMQ); конкретный вендор считается заменяемым и фиксируется в ADR.
  - Для хранения транзакционных данных используется реляционная СУБД (OLTP), для высокообъёмных событий и метрик — специализированные хранилища (Time-series/Columnar/Object Storage).

- **Границы ответственности архитектурного решения**
  - Архитектурный кейс определяет целевую логическую и физическую архитектуру, принципы разбиения на микросервисы, интерфейсы взаимодействия и требования к инфраструктуре.
  - Операционные процессы (ITIL/ITSM, процессы эксплуатации, регламенты NOC/Service Desk) считаются внешними по отношению к данному описанию и формализуются только через требования к интерфейсам и событиям (alarms, notifications, audit).
  - Выбор конкретных продуктов и, при необходимости, проприетарных решений (SBC, медиашлюзы, системы записи) оформляется отдельными технологическими решениями и не детализируется в рамках базового архитектурного описания, если не оговорено в ADR.

Все допущения и границы ответственности являются частью архитектурного контракта проекта и используются как основа для интерпретации требований, сценариев и проектных решений во всех последующих разделах.
